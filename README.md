# Intants AI Voice Interview Platform

A voice-first AI interview platform. Candidates log in, pick a job role, and talk to a realistic real-time avatar interviewer in their language (EN/HI/TE Day-1, expanding to 22 Indian languages), and receive a structured scorecard at the end of a ~10-minute session.

**Status:** Demo product **built and running** (Phase 1+). Four FastAPI microservices + a LiveKit real-time worker + a React/Vite frontend, all runnable locally against cloud demo infrastructure. The production (Tier-2 AWS Mumbai) path is the same code, env-swappable.

### What's built

- **Real-time avatar interview** over LiveKit (WebRTC) — Tavus avatar lip-syncs Sarvam TTS in echo mode; Simli also supported; `none` = voice-only.
- **6 avatars** (3 male / 3 female) with an in-app **avatar picker** (`GET /api/avatars`).
- **10-question interview window** (exactly 10 candidate answers) with a ~12-minute session cost cap.
- **Scoring + PDF scorecard** generated by `feedback_billing` after the session.
- **Admin / analytics dashboard** served by `admin_ops` (:8004).
- **Pluggable auth** — local register/login/refresh/logout/me, Google SSO, Naipunyam SSO, DPDP consent, jobs + resume upload.
- **Multilingual** — EN/HI/TE interview voice + EN/HI/TE candidate-journey UI with a language switcher; history + resume of past sessions.

## Running locally (dev)

> **DB (Neon / Prisma Postgres) and Redis (Upstash) are cloud** — nothing to start locally.
> Each service reads its own `services/<name>/.env` (see `docs/PROCUREMENT.md` for credentials).

The demo runs **6 processes**. Note that `interview_core` runs as **two** of them — the FastAPI
**API** and a separate **LiveKit worker** (the real-time avatar + voice engine). They *must* be
separate processes: the worker calls `cli.run_app(WorkerOptions(...))`, which takes over its own
process and spawns a subprocess per interview — it cannot run inside the uvicorn API. They also
scale on different axes (API by HTTP traffic, worker by concurrent interviews).

### One command (Windows)

```powershell
# from repo root — launches all 6 in separate windows so you can watch each log
.\dev-up.ps1
```

Then open **http://localhost:5174**.

### Manual (one PowerShell terminal each)

```powershell
# 1. data_gateway — auth (:8002)            [uses the in-project .venv]
cd services\data_gateway
.\.venv\Scripts\python.exe -m uvicorn app.main:app --host 0.0.0.0 --port 8002

# 2. interview_core — API (:8001)           [uses the in-project .venv]
cd services\interview_core
.\.venv\Scripts\python.exe -m uvicorn app.main:app --host 0.0.0.0 --port 8001

# 3. feedback_billing — scoring (:8003)     [uses the in-project .venv]
cd services\feedback_billing
.\.venv\Scripts\python.exe -m uvicorn app.main:app --host 0.0.0.0 --port 8003

# 4. admin_ops — admin/analytics dashboard (:8004)   [uses the in-project .venv]
cd services\admin_ops
.\.venv\Scripts\python.exe -m uvicorn app.main:app --host 0.0.0.0 --port 8004

# 5. interview_core — LiveKit worker (no HTTP port; the avatar/voice engine)
#    PYTHONUTF8=1 stops the rich-console UnicodeEncodeError log spam on Windows.
cd services\interview_core
$env:PYTHONUTF8="1"
.\.venv\Scripts\python.exe -m app.worker.interview_worker dev

# 6. web — frontend (:5174)
cd web
npm run dev
```

**Gotchas**
- **All four services** run the in-project **`.venv` python directly** (`py -3.12 -m venv .venv`
  then `.\.venv\Scripts\python.exe -m pip install -r requirements.txt`). Do **not** `poetry install`
  interview_core (its `poetry.lock` is out of sync with the pip-managed LiveKit stack; a sync would
  uninstall working packages). `shared` is made importable via a `.pth` file dropped in each `.venv`.
- **Start the worker before** beginning an interview — LiveKit auto-dispatch only assigns a worker
  to rooms created while it is running.
- **`admin_ops` (:8004)** is started by `dev-up.ps1` and serves the admin/analytics dashboard.
- Liveness: `GET /health/live` on `:8001` / `:8002` / `:8003` / `:8004`.

Each service is **independently deployable** — its `.env` lives next to it, only the variables that
service needs are in it.
